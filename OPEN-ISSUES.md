## 議題一：Reviewer 的審核時序問題 ✅ 已決議：方案 A

### 背景

`relay.py` 目前採**並行**呼叫三個 AI 子進程的架構。這導致「Reviewer」角色在同一輪中看到的，是**修改前**的代碼版本，無法即時審核「Executor」在該輪的實際產出。

### 目前提案

| 方案                             | 提案者             | 說明                                                                                                                   | 優點                             | 缺點                                                                                |
| -------------------------------- | ------------------ | ---------------------------------------------------------------------------------------------------------------------- | -------------------------------- | ----------------------------------------------------------------------------------- |
| **方案 A：延後一輪審核制**       | [gemini]           | Reviewer 在**下一輪**對前一輪的 Diff 進行回溯審核，SOP 明定此為正式流程                                                | 符合目前並行技術現實，不需改架構 | 問題在隔輪才被指出，修復延遲一個循環                                                |
| **方案 B：系統層強制排序**       | [claude] / [codex] | 在 `relay.py` 呈現層強制「Executor 先、Reviewer 後」的輸出順序，並在 Prompt 中要求 Reviewer 僅針對當輪「報告區塊」評價 | 同輪就可察覺問題                 | 輸出順序≠執行順序，Reviewer 拿到的仍是執行前版本；需大幅修改 relay 架構才能真正做到 |
| **方案 C：序列化執行（未提案）** | —                  | 將三個 AI 改為**序列**呼叫：Executor → 輸出 → Reviewer 讀取輸出 → 審核                                                 | Reviewer 能真正看到最新產出      | 整體回應時間大幅拉長；架構需重寫                                                    |

### 需要 Human 決定

- 短期採用哪個方案？
- 「延後一輪審核」是否應正式寫入 SOP 作為規範？

### [human] 傾向

> **建議方案 A**。並行是核心價值，串聯會把每輪回應時間拉長 3 倍。方案 B 本質上 Reviewer 拿到的仍是執行前版本，只是排版好看而已。「隔一輪發現問題」符合自然討論節奏，不算缺點。建議在 SOP 明確寫入「Reviewer 採延後一輪審核制」即可。

## 議題二：寫入安全與 `bypassPermissions` 機制 ✅ 已決議：方案 C

### 背景

目前 `run_claude_cli.js` 使用 `--dangerously-skip-permissions`（即 `bypassPermissions`），跳過了 Human 的逐步確認。SOP v0.2 雖已聲明此為「現況 Convention」，但尚未決定是否在程式層進行約束。

### 目前提案

| 方案                                | 說明                                                                        | 風險                               |
| ----------------------------------- | --------------------------------------------------------------------------- | ---------------------------------- |
| **方案 A：僅靠 SOP 條款約束**       | 維持現狀，透過角色規範與 SOP 來限制 AI 的寫入行為                           | AI 若角色偏移，SOP 無強制力        |
| **方案 B：relay.py 攔截非唯讀指令** | 在 `relay.py` 中加入白名單或攔截器，偵測到寫入/刪除類指令時暫停並通知 Human | 攔截邏輯難以窮舉，可能誤擋合法操作 |
| **方案 C：分模式執行**              | 新增 `RELAY_MODE=readonly` 環境變數；唯讀模式下自動關閉 `bypassPermissions` | 需要每次手動切換，操作成本提高     |

### 需要 Human 決定

- 是否需要在程式層加入防護？
- 若是，採用哪種攔截策略？

### [human] 傾向

> **建議方案 C（分模式）**。純 SOP 約束沒有強制力（方案 A），窮舉攔截幾乎不可能且誤判多（方案 B）。加一個 `RELAY_MODE=readonly|full` 環境變數，`readonly` 時移除 Claude 的 `--permission-mode bypassPermissions`、Gemini 的 `--yolo`、Codex 的 `--dangerously-bypass-approvals-and-sandbox`。預設值建議為 `readonly`，需要寫入時才手動切換 `RELAY_MODE=full`。改動量約 20 行，分布在 4 個 wrapper 中。

## 議題三：軟性超時告警（Soft Timeout） ✅ 已決議：方案 C（移至 Backlog）

### 背景

目前只有「硬超時」（`RELAY_TIMEOUT_SEC`，超過後強制終止子進程）。[codex] 建議增加一個**軟告警**機制：若某 AI 超過一定時間未回應，主動通知 Human，而非等到硬超時才被動終止。

### 目前提案

| 方案                     | 說明                                                                                 |
| ------------------------ | ------------------------------------------------------------------------------------ |
| **方案 A：不加軟告警**   | 維持現狀，依賴硬超時即可，避免增加複雜度                                             |
| **方案 B：固定閾值告警** | 若某 AI 超過 N 秒（如 120s）未輸出，在終端機印出警告訊息，Human 可選擇等待或手動中斷 |
| **方案 C：動態閾值告警** | 軟告警閾值 = `RELAY_TIMEOUT_SEC × 0.5`，與超時設定連動                               |

### 需要 Human 決定

- 是否需要軟告警？
- 若是，告警閾值為固定值或動態比例？
- 告警後系統繼續等待，還是讓 Human 選擇？

### [human] 傾向

> **建議方案 C（動態閾值），但優先級降低，放 Backlog**。目前 `print_round_summary` 已顯示各 AI 耗時，Human 能看到。如果要做，動態 50% 比固定 120s 好（因為 TIMEOUT 可調，固定值會不匹配）。實作可在 `ask_all()` 用 `concurrent.futures.wait(timeout=TIMEOUT*0.5)` 做第一次檢查，未完成的印警告。不急。

## 議題四：`.env` 自動載入支援 ✅ 已決議：方案 B

### 背景

SOP v0.2 第 2 節強調「集中管理環境變數」，目前代碼透過 `os.environ.get()` 讀取，但**不會自動載入 `.env` 檔案**。若使用者建立 `.env` 而未事先手動 `export`/`set`，環境變數不會生效。

### 目前狀態

- SOP 尚未明確說明此限制（[claude] 已在代碼審查中指出，建議 SOP 補注）
- 代碼層未引入 `python-dotenv`

### 選項

| 選項                               | 說明                                                                                              | 成本                                        |
| ---------------------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| **選項 A：維持現狀，SOP 補注說明** | 在 SOP 第 2 節加一行說明「目前需手動 export，非自動載入」                                         | 零程式改動                                  |
| **選項 B：加入 python-dotenv**     | `relay.py` 頂部加 `from dotenv import load_dotenv; load_dotenv()`；需 `pip install python-dotenv` | 新增一個外部依賴，需更新 `requirements.txt` |
| **選項 C：手工解析 `.env`**        | 自行撰寫簡單的 `.env` 解析器，不引入外部依賴                                                      | 程式改動較多，且容易有邊界案例              |

### 需要 Human 決定

- 是否希望支援 `.env` 自動載入？
- 若是，是否接受 `python-dotenv` 作為新依賴？

### [human] 傾向

> **建議方案 B（python-dotenv）**。這是 Python 生態最成熟的 `.env` 方案，零風險。手工解析（方案 C）會踩引號、多行值、註解等邊界坑。實作只需在 `relay.py` 頂部加 `from dotenv import load_dotenv; load_dotenv()`，各 wrapper 不需改（因為是子進程，relay 的 `_non_interactive_env()` 已用 `os.environ.copy()` 傳遞）。需加 `requirements.txt`。

## 議題五：`run_claude_cli.js` 與 `run_claude_cli.py` 並存 ✅ 已決議：選項 B

### 背景

目前目錄中同時存在 `run_claude_cli.js`（Node.js，呼叫 `claude` CLI）與 `run_claude_cli.py`（Python wrapper）兩支功能相近的腳本。

- `relay.py` 目前呼叫的是 `.py` 版本（確認自代碼審查）
- `.js` 版本的作用、使用時機、與 `.py` 版本的差異，目前未在 SOP 或 README 中說明

### 潛在風險

- 維護者不清楚兩者的關係，可能造成修改其中一支而遺漏另一支
- 若未來有人改用 `.js` 版，行為可能與 `.py` 版不一致

### 選項

| 選項                                 | 說明                                     |
| ------------------------------------ | ---------------------------------------- |
| **選項 A：在 SOP / README 補充說明** | 說明兩者的用途分工與差異，不做刪除       |
| **選項 B：統一僅保留一支**           | 決定主力版本，刪除或封存另一支           |
| **選項 C：保留兩支但加測試**         | 兩支都維護，並加入自動化測試確保行為一致 |

### 需要 Human 決定

- 兩支腳本的預期角色為何？
- 是否應統一為單一腳本？

### [human] 傾向

> **建議選項 B（刪除 `.js` 版，僅保留 `.py` 版）**。`relay.py` 已呼叫 `.py` 版本，`.js` 版目前完全未被使用。且 `.js` 的 `SYSTEM_PROMPT` 比 `.py` 版簡陋許多（缺少角色隔離指令），兩者行為已不一致。保留只會造成維護遺漏風險（如這次的亂碼修復就是案例）。建議刪除或重命名為 `.deprecated`。
